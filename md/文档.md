# 基于LangGraph的智能旅行助手 - 实战报告

## 一、实战内容

### 1. 项目概述
基于LangGraph框架构建多Agent协作的旅行规划系统，实现从天气查询、景点推荐、行程规划到预算估算的全流程自动化旅行助手。

### 2. 核心实现模块

#### （1）状态管理（states.py）
- 定义`PlanningStatus`枚举：标识5个规划阶段（INIT → WEATHER → ATTRACTION → ITINERARY → BUDGET → DONE）
- 使用Pydantic定义`TravelInfo`数据模型，包含目的地、日期、用户偏好及各Agent输出字段

#### （2）Agent基类（base_agent.py）
- 封装OpenAI API调用逻辑，支持异步/同步双模式
- 实现JSON解析方法，通过正则提取大模型返回的结构化数据
- 配置温度参数（0.2）、超时控制（30s）和重试机制（2次）

#### （3）四大功能Agent
**天气Agent（weather_agent.py）**
- 集成聚合数据API获取真实天气预报
- 实现异步HTTP请求（run_in_executor避免阻塞事件循环）
- 提供降级策略：API失败时根据季节生成模拟天气数据
- 智能建议生成：根据降雨、高温、低温等情况给出出行建议

**景点Agent（attraction_agent.py）**
- 结合天气状况和用户偏好推荐4个景点
- 规范化输出格式：名称、类型、描述、最佳游览时间、交通时长、建议
- 降级方案：API异常时返回文化博物馆、城市公园、地标、美食街等默认景点

**行程Agent（itinerary_agent.py）**
- 基于景点列表生成每日详细行程（上午/下午/晚上）
- 整合交通建议和餐饮推荐
- 支持灵活字段映射（兼容days/schedule等不同键名）

**预算Agent（budget_agent.py）**
- 根据旅行天数、预算等级（high/mid-range/economy）估算费用
- 分类统计：住宿(40%)、餐饮(25%)、交通(15%)、门票(15%)、杂费(5%)
- 动态调整基准价格（高档1200元/天，中档700元/天）

#### （4）工作流编排（main.py）
- 使用`StateGraph`构建线性工作流：START → weather_analysis → attraction_recommendation → itinerary_planning → budget_estimate
- 实现命令行参数解析，支持交互式输入缺失参数
- 异步执行图编译结果（ainvoke），最终输出JSON格式旅行计划

### 3. 技术栈
- **框架**：LangGraph 0.3+、LangChain 0.3+
- **模型**：GPT-3.5-turbo（OpenAI）
- **数据验证**：Pydantic 2.x
- **异步编程**：asyncio + aiohttp
- **外部API**：聚合数据天气接口

## 二、遇到的问题

### 问题1：异步HTTP请求与事件循环冲突
**现象**：在异步方法中直接调用`requests.get()`导致阻塞或RuntimeError  
**原因**：requests是同步库，在asyncio事件循环中会阻塞其他协程

### 解决1：使用线程池执行同步请求(weather_agent.py,50-53行)
将同步的requests调用放入线程池，避免阻塞事件循环
loop = asyncio.get_event_loop()
response = await loop.run_in_executor(None, self._sync_request, params)



### 问题2：大模型返回非标准JSON格式(base_agent.py,53-58行)
**现象**：模型输出包含Markdown标记、额外文本说明，导致`json.loads()`解析失败  
### 解决2：正则表达式提取JSON
先尝试提取花括号/方括号包裹的内容，再解析为JSON对象



### 问题3：状态传递中的数据完整性(weather_agent.py,139-147行)
**现象**：后续Agent访问前置Agent的输出时出现KeyError或NoneType错误  
**原因**：异常降级时未保证返回数据结构一致
### 解决3：统一降级数据结构
每个Agent的fallback方法都返回完整字段结构，例如：
确保无论成功或失败，下游Agent都能安全访问字段



### 问题4：相对导入路径错误
**现象**：运行时报错`ImportError: attempted relative import with no known parent package`  
**原因**：在agents子模块中使用`from ..states import TravelInfo`导致路径解析失败
### 解决4：改用绝对导入(项目中所有py文件)
在项目根目录运行时，使用相对于项目根的绝对导入路径



### 问题5：API密钥管理混乱
**现象**：硬编码API Key存在安全风险，且不同服务（OpenAI、天气API）密钥混用  
**影响**：代码可维护性差，部署时需手动修改多处
### 解决5：环境变量管理密钥
优先读取环境变量，未设置时使用默认值（仅用于开发测试）



### 问题6：依赖版本兼容性
**现象**：`langchain.chat_models.init_chat_model`在新版本中API变更  
**报错**：`ModuleNotFoundError: No module named 'langchain.chat_models'`
### 解决6：锁定依赖版本并验证API
在requirements.txt中指定最低版本：
查阅官方文档确认`init_chat_model`的正确用法，必要时降级到稳定版本

**现象**：字段名不匹配导致数据结构不一致(天气、景点、行程、预算)
**原因**：不同模块使用不同的字段名，导致数据结构不一致。前端对话模型后,查询的字段名与各Agent返回的实际字段名不匹配
